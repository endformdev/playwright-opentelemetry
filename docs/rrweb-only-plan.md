# rrweb-only Trace Viewer Plan

## Goal

Replace Playwright trace screenshot extraction with rrweb recording and replay. The trace viewer should no longer depend on screenshot manifests, screenshot ZIPs, virtual screenshot URLs, or the service worker screenshot extraction path.

The end state is:

- The fixture injects rrweb into browser pages during Playwright test execution.
- The reporter stores rrweb recordings alongside OTLP span data in local trace ZIPs.
- The reporter uploads rrweb recordings to the Trace API when `playwrightTraceApiEndpoint` is configured.
- The Trace API stores and serves rrweb recording ZIPs at trace-scoped paths.
- The viewer loads rrweb recording data directly, without a service worker.
- The viewer renders a persistent full replay panel on the right and a bounded rrweb filmstrip in the timeline.
- Screenshot extraction, `screenshots.zip`, screenshot service-worker endpoints, screenshot UI types, and screenshot docs are removed.

## Key Assumptions

- rrweb traces are generated by this package's fixture, not by parsing Playwright's trace ZIP.
- rrweb replay data is trusted to the same degree as the test artifacts users already load.
- Canvas replay is desired and will use `UNSAFE_replayCanvas: true` in the viewer.
- The hosted viewer should run on a dedicated origin with no sensitive cookies or privileged local state.
- Existing screenshot-based traces do not need backward compatibility unless a separate migration requirement is added.
- The first implementation can load `rrweb.zip` as one artifact; segmented API endpoints can be added later if artifact size becomes an issue.

## Target Artifact Format

### Local `*-pw-otel.zip`

```text
{file.spec}:{linenumber}-{testId}-pw-otel.zip
- traces/
  - playwright-opentelemetry.json
  - playwright-browser.json
- rrweb/
  - manifest.json
  - recordings/
    - page-1/
      - 00000.json
      - 00001.json
    - page-2/
      - 00000.json
```

### `rrweb/manifest.json`

```json
{
  "version": 1,
  "recordings": [
    {
      "id": "page-1",
      "pageId": "page-1",
      "startTime": 1766927492000,
      "endTime": 1766927493000,
      "eventCount": 240,
      "initialUrl": "https://example.test/checkout",
      "segments": [
        {
          "file": "rrweb/recordings/page-1/00000.json",
          "startTime": 1766927492000,
          "endTime": 1766927492500,
          "eventCount": 120,
          "hasFullSnapshot": true
        }
      ]
    }
  ],
  "recordingOptions": {
    "recordCanvas": true,
    "recordCrossOriginIframes": true,
    "collectFonts": true,
    "inlineImages": true,
    "checkoutEveryNms": 5000
  }
}
```

### Standalone `rrweb.zip`

The uploaded API artifact uses the same `rrweb/manifest.json` and `rrweb/recordings/**` layout inside a ZIP.

The local full trace ZIP embeds the rrweb directory directly. The remote API serves the rrweb directory as a dedicated `rrweb.zip` object.

### Recorder Injection

Add a fixture-side recorder module, for example:

```text
reporter/src/fixture/rrweb-recorder.ts
reporter/src/fixture/rrweb-recorder-browser.ts
```

Responsibilities:

- Build a browser-executable recorder script that imports or bundles `@rrweb/record`.
- Register an exposed binding before `addInitScript`.
- Start rrweb recording in every document as early as Playwright allows.
- Buffer events in the page and flush to Node periodically.
- Flush on `pagehide`, `visibilitychange`, and teardown.
- Stop recording at fixture teardown.
- Attach the final rrweb artifact metadata/body to the Playwright `testInfo` so the reporter can consume it.

Use `browserContext.addInitScript` from `reporter/src/fixture/playwright-opentelemetry-fixture.ts` rather than only instrumenting the initial `page` fixture. This is needed for navigations, popups, and pages created by `context.newPage()`.

Use `browserContext.exposeBinding` for event transport:

```ts
await context.exposeBinding("__playwrightOtelRrwebEmit", async (source, payload) => {
  // source.page() identifies the Playwright page.
  // payload contains events, metadata, flush reason, and document id.
});
```

Payload shape:

```ts
interface RrwebFlushPayload {
  documentId: string;
  href: string;
  reason: "interval" | "checkout" | "pagehide" | "teardown" | "error";
  events: eventWithTime[];
  warnings?: string[];
}
```

The Node side groups payloads by Playwright `Page`, fixture `pageId`, and `documentId`. A full navigation can create a new rrweb document stream while still belonging to the same Playwright page ID.

Use a page identity mapping shared with `BrowserPageTracker` so rrweb recordings can be related to browser spans:

```ts
browserPageTracker.registerPage(page);
const pageId = browserPageTracker.getPageId(page);
```

If `BrowserPageTracker` does not expose page IDs today, add a minimal `getPageId(page: Page): string` method.

### Browser Recorder Behavior

The browser script should call rrweb `record` with high-fidelity options:

```ts
record({
  emit(event, isCheckout) {
    buffer.push(event);
    if (isCheckout) markCurrentSegmentAsCheckout();
    scheduleFlush();
  },
  recordCanvas: true,
  recordCrossOriginIframes: true,
  collectFonts: true,
  inlineImages: true,
  checkoutEveryNms: 5000,
  recordAfter: "DOMContentLoaded",
  errorHandler(error) {
    flushWarning(error);
  }
});
```

Notes:

- `recordAfter: "DOMContentLoaded"` is a reasonable compromise. Starting after `load` misses earlier app rendering; starting before DOMContentLoaded is not supported in the same way by rrweb.
- rrweb emits Unix millisecond timestamps. These can align directly to span/test timestamps.
- Cross-origin iframe recording is best-effort. rrweb requires injection in child frames, and browser security restrictions still apply.
- The recorder must fail open. If rrweb throws, it should record a warning and allow the Playwright test to continue.

### New rrweb Loader Modules

Add:

```text

```

Suggested types:

```ts
import type { eventWithTime } from "@rrweb/types";

export interface RrwebTraceManifest {
  version: 1;
  recordings: RrwebRecordingManifest[];
  recordingOptions?: Record<string, unknown>;
}

export interface RrwebRecordingManifest {
  id: string;
  pageId: string;
  startTime: number;
  endTime: number;
  eventCount: number;
  initialUrl?: string;
  segments: RrwebSegmentManifest[];
}

export interface RrwebSegmentManifest {
  file: string;
  startTime: number;
  endTime: number;
  eventCount: number;
  hasFullSnapshot: boolean;
}

export interface RrwebRecording {
  id: string;
  pageId: string;
  startTime: number;
  endTime: number;
  initialUrl?: string;
  events: eventWithTime[];
}

export interface RrwebTrace {
  recordings: RrwebRecording[];
}
```

Initial loader behavior:

- Open local full trace ZIP with `ZipReader`.
- Parse all `traces/*.json` into OTLP export data.
- Parse `rrweb/manifest.json`.
- Read all listed rrweb segment JSON files.
- Return `RrwebTrace` with events sorted by timestamp per recording.
- Validate that every recording has at least one rrweb `Meta` event and one `FullSnapshot` event before trying to create a replayer.

Remote API behavior:

- Fetch `{baseUrl}/traces`.
- Fetch `{baseUrl}/rrweb.zip`.
- If `rrweb.zip` returns 404, return an empty rrweb trace and show a no-recording state.
- Parse the ZIP in app code.

Remote ZIP behavior:

- Fetch the full trace ZIP URL.
- Parse OTLP and rrweb directly from the ZIP in app code.

This removes the need to post messages to a service worker or to synthesize `/screenshots/*` URLs.

If ZIP parsing or JSON decoding becomes expensive, add a dedicated Web Worker for parsing. Do not keep or recreate the service worker unless the viewer again needs network interception or synthetic URLs.

### rrweb Replay Components

Add:

```text
trace-viewer/src/trace-viewer/rrweb/RrwebReplayPreview.tsx
trace-viewer/src/trace-viewer/rrweb/RrwebThumbnail.tsx
trace-viewer/src/trace-viewer/rrweb/useRrwebReplayer.ts
```

`RrwebReplayPreview`:

- Owns one persistent `Replayer` instance.
- Creates the replayer once for the selected recording.
- Calls `replayer.pause(offsetMs)` when `displayTimeMs` changes.
- Throttles seeking with `requestAnimationFrame`.
- Uses `UNSAFE_replayCanvas: true`.
- Uses `mouseTail: false` unless cursor visualization is desired.
- Calls `replayer.destroy()` on cleanup.

Offset calculation:

```ts
const absoluteTimeMs = testStartTimeMs + displayTimeMs;
const offsetMs = absoluteTimeMs - recording.startTime;
replayer.pause(Math.max(0, offsetMs));
```

Recording selection:

- If one recording exists, use it.
- If multiple recordings exist, prefer the recording whose time range contains `absoluteTimeMs`.
- If multiple recordings contain the time, prefer the page associated with the active browser span if that association is available.
- Otherwise choose the first recording by start time.

`RrwebThumbnail`:

- Owns one small paused `Replayer` instance.
- Seeks to a fixed timestamp.
- Does not update on every hover movement.
- Destroys and recreates only when assigned frame changes.

### Filmstrip Performance Strategy

The filmstrip should not create unbounded replayers.

Use one of these strategies for the first implementation:

- Render only visible slots and cap `slotCount` to 6.
- Or render only visible slots and maintain a pool of 4 to 6 thumbnail components.

Recommended first version:

```ts
const MAX_RRWEB_FILMSTRIP_REPLAYERS = 6;
setSlotCount(Math.min(MAX_RRWEB_FILMSTRIP_REPLAYERS, computedSlotCount));
```

Keep the right-side preview separate and persistent. Do not share its replayer with the filmstrip.

Expected cost:

- One full preview replayer should be acceptable.
- Four to six thumbnail replayers should be acceptable for moderate DOM traces.
- Canvas-heavy traces may still be expensive. If this becomes a problem, add thumbnail degradation such as fewer slots or canvas disabled only in thumbnails.

## Security Model

Because canvas replay requires `UNSAFE_replayCanvas: true`, the viewer must treat rrweb trace data as trusted executable replay data.

Required guidance:

- Do not load arbitrary public rrweb trace ZIPs in a viewer origin with sensitive cookies.
- Host the public viewer on a dedicated origin.
- Keep the Trace API read endpoints scoped to trace artifacts only.
- Avoid storing auth tokens in localStorage/sessionStorage on the viewer origin.
- If untrusted uploads are ever supported, disable canvas replay or replay in a separate isolated origin.
